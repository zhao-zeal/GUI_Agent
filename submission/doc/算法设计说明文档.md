# 移动端 GUI Agent 算法设计说明文档

## 1. 赛题理解

本赛题要求设计一个基于多模态大模型的移动端 GUI Agent。系统输入包括用户自然语言任务和当前安卓手机截图；系统输出当前步骤需要执行的下一步动作。动作空间包括 CLICK、TYPE、SCROLL、OPEN、COMPLETE。Agent 需要在多轮调用中逐步完成任务，并保证每一步输出满足评测接口要求。

本方案重点解决三个问题：

1. **多模态理解**：结合用户任务、当前截图和历史动作，判断当前界面状态。
2. **下一步动作决策**：将复杂任务拆解为一步一步的 GUI 操作。
3. **稳定格式解析**：无论模型输出 JSON、函数调用格式还是自然语言描述，都尽量解析为评测器要求的标准 `AgentOutput`。

## 2. 总体架构

本方案采用“VLM 决策 + 强约束提示词 + 稳健解析器 + 保守兜底策略”的架构。

```text
AgentInput
  ├── 用户任务 instruction
  ├── 当前截图 current_image
  ├── 当前步数 step_count
  └── 历史动作 history_actions
        ↓
Prompt Builder
        ↓
VLM API 调用：BaseAgent._call_api
        ↓
Raw Model Output
        ↓
Action Parser
        ↓
Schema Sanitizer
        ↓
AgentOutput(action, parameters)
```

## 3. 核心模块设计

### 3.1 Prompt Builder

Prompt Builder 会将任务、截图尺寸、历史动作、动作空间、坐标归一化规则和决策原则统一组织成给多模态模型的输入。提示词明确要求模型只输出 JSON，格式如下：

```json
{
  "thought": "中文简要说明判断依据",
  "action": "CLICK|TYPE|SCROLL|OPEN|COMPLETE",
  "parameters": {}
}
```

坐标采用 0 到 1000 的归一化坐标系。模型如果根据原图估计像素坐标 `(x_pixel, y_pixel)`，需要转换为：

```text
x_norm = round(x_pixel / image_width * 1000)
y_norm = round(y_pixel / image_height * 1000)
```

### 3.2 图像压缩

移动端截图通常较大，直接输入会增加 token 和接口成本。方案在不改变相对坐标的前提下，将截图最长边缩放至 1440 像素以内，再进行 base64 编码。由于输出使用归一化坐标，缩放不会影响评测坐标体系。

### 3.3 多轮历史利用

Agent 会读取 `history_actions`，将最近若干步动作写入提示词，提醒模型避免重复点击同一无效位置。如果历史动作显示界面没有明显推进，模型应选择替代控件或滚动。

### 3.4 Action Parser

模型原始输出不一定完全符合要求，可能出现：

- 标准 JSON：`{"action":"CLICK","parameters":{"point":[500,300]}}`
- 函数调用：`click(point='<point>500 300</point>')`
- 题面示例：`CLICK:[[100, 200]]`
- 自然语言混合输出：`Action: CLICK:[[100,200]]`

因此实现了 `utils/action_parser.py`，支持宽松解析并统一转换为标准动作参数。

标准输出包括：

```python
CLICK    -> {"point": [x, y]}
SCROLL   -> {"start_point": [x1, y1], "end_point": [x2, y2]}
TYPE     -> {"text": "内容", "content": "内容"}
OPEN     -> {"app_name": "应用名"}
COMPLETE -> {}
```

其中 TYPE 同时保留 `text` 和 `content`，是为了兼容题面参数名与公开测试代码之间可能存在的不一致。当前公开测试代码读取的是 `text` 字段；题面截图中参数名为 `content`。保留两个字段可以提高兼容性。

### 3.5 保守兜底策略

如果 API 调用失败或模型输出无法解析，Agent 不返回非法格式，而是执行保守策略：

1. 前两步如果能从任务中推断目标 App，则输出 `OPEN`。
2. 超过最大步数附近时输出 `COMPLETE`，避免无限循环。
3. 其他情况输出默认上滑 `SCROLL`，尽量探索更多界面内容。

## 4. 决策策略

Agent 的主要策略如下：

1. **应用未打开时优先 OPEN**：如果任务明确提到美团、微信、支付宝、携程等 App，优先直接打开 App。
2. **弹窗优先处理**：遇到权限、隐私、升级、广告等弹窗，优先点击同意、允许、跳过、关闭、稍后。
3. **表单分步填写**：搜索或填写表单时，先点击输入框，再输入内容，再点击搜索或确认按钮。
4. **坐标点击中心点**：对按钮、列表项、输入框等可点击区域，选择控件中心点，减少边界误差。
5. **滚动探索**：目标内容不在当前屏幕时，通过 SCROLL 查找更多选项。
6. **完成条件严格**：只有当前界面已经能证明任务完成时才输出 COMPLETE。

## 5. 创新点

### 5.1 强约束 JSON 输出 + 宽松解析

仅依赖 Prompt 无法保证模型严格按格式输出。因此方案采用“双保险”：先用强约束提示词要求 JSON，再用宽松解析器兼容多种输出格式。这样可以减少因格式错误导致的 0 分。

### 5.2 归一化坐标显式教学

提示词中明确给出像素坐标到归一化坐标的转换公式，降低模型输出实际像素坐标的概率。

### 5.3 任务意图到 App 的兜底映射

当模型调用失败时，系统可以根据任务关键词推断目标 App。例如外卖映射到美团，酒店/机票映射到携程，导航映射到高德地图。这能让部分初始状态用例仍有机会进入正确 App。

### 5.4 历史动作防重复

Prompt 中注入历史动作，使模型知道之前已经做过什么，减少重复点击和重复输入。

## 6. 复杂度与资源占用

本方案主要依赖一次多模态模型调用，解析器为正则和 JSON 解析，计算复杂度为 O(n)，其中 n 为模型输出文本长度。额外内存开销主要来自压缩后的截图和少量历史动作记录。提交包不包含大模型权重和大型依赖，满足 20 MB 限制。

## 8. 本地测试方式

将本提交包中的 `src` 目录与主办方提供的 `test_data/offline` 放在同一工作目录下，配置好 `VLM_API_KEY` 后运行：

```bash
cd src
python test_runner.py --data_dir ./test_data/offline
```

如果本地测试代码中的 TYPE 参数确认为 `content` 而非 `text`，可在 `agent.py` 的 `_sanitize` 方法和 `utils/action_parser.py` 的 TYPE 输出处将主字段调整为 `content`。当前实现保留了两个字段以兼容两种情况。
